作为专业的数仓开发 Agent，在生成代码和进行架构设计时，必须严格遵守以下分层规范与防坑铁律。

## 1. ODS 层 (Operational Data Store) 贴源数据层
* **核心定位：** 保持与业务源系统数据结构绝对一致，原汁原味，提供数据追溯的根基。
* **命名规范：** `ods_{业务源系统库名}_{源表名}_{全量/增量标识}`（例如：`ods_erp_trade_order_df`）。
* **同步策略：** 严格按照业务系统的生成或更新时间，按天进行分区存储，保留历史快照。
* **开发底线：** 仅做数据类型的统一映射和极度敏感字段的简单掩码，严禁在此层进行任何多表 Join、过滤或业务逻辑计算。

## 2. DWD 层 (Data Warehouse Detail) 数据明细层
* **核心定位：** 构建企业级一致性维度建模，对 ODS 数据进行清洗、规范化和维度退化，解决“数据脏乱差”。
* **命名规范：** `dwd_{主题域}_{业务过程}_{事实表类型标识}`（例如：`dwd_trade_order_detail_df`）。
* **清洗规范：** 强制处理空值（如 `NULL` 替换为默认值或 `0`）、标准化枚举值（如男女统一为 `M/F`）、统一数据类型和单位（如金额统一精确到两位小数）。
* **开发底线：** 必须使用 CTE（`WITH` 语句）构建代码逻辑。对数据流进行唯一性校验以去重。事实表必须能直接关联到标准维度表。

## 3. DIM 层 (Dimension) 公共维度层
* **核心定位：** 构建企业一致性的维度视图（如：时间、地区、用户、商品），消除数据孤岛，保证各业务线分析口径一致。
* **命名规范：** `dim_{主题域/业务实体}_{全量/拉链标识}`（例如：`dim_pub_date_df`、`dim_trade_user_zipper`）。
* **开发底线：** 针对变化属性熟练使用拉链表记录历史轨迹。对于树状层级必须在 DIM 层打平成宽表，严禁下游写递归。必须预埋默认未知记录（如 `id = -1`）。

## 4. DWS 层 (Data Warehouse Summary) 数据汇总层
* **核心定位：** 以分析的主题对象为基准，进行轻度聚合计算，沉淀可复用的公共指标。
* **命名规范：** `dws_{主题域}_{分析实体粒度}_{时间周期}`（例如：`dws_trade_user_order_1d`）。
* **聚合规范：** 涵盖高频的业务分析维度与时间周期，通过构建宽表减少下游的重复计算。
* **开发底线：** 强依赖 DWD 层和 DIM 层，严禁跨层直读 ODS。派生指标计算逻辑必须在此层收口。

## 5. ADS 层 (Application Data Service) 应用数据层
* **核心定位：** 直接面向业务的定制化需求，为 BI 报表、大屏看板等提供开箱即用的数据结果。
* **命名规范：** `ads_{业务应用场景}_{报表/结果集名称}`（例如：`ads_bi_dashboard_sales_kanban`）。
* **性能规范：** 数据结构必须极度扁平化（大宽表或 KV 结构），不包含复杂 Join 或聚合，确保毫秒级响应。
* **开发底线：** 数据必须来自 DWS 或 DWD。按需加工，不强调复用性。

## 6. TMP 层 (Temporary/Staging) 临时中间层
* **核心定位：** 存放复杂数据加工过程中的中间阶段结果，分担超级 SQL 的计算压力或实现复用。
* **命名规范：** `tmp_{业务过程}_{结果描述}_{开发者/时间戳}`（例如：`tmp_trade_order_user_rank_202310`）。
* **开发底线：** 阅后即焚，建表必须带极短生命周期（TTL 1~7天）。严禁其他层级（特别是 ADS）直接依赖 TMP 出核心报表。

## 7. 全局通用与防坑规范 (Agent 铁律)
* **注释与命名：** 表与字段注释率必须达到 100%。同义字段在各层物理命名绝对一致。
* **防爆与性能：** 所有事实表必须带时间分区，`WHERE` 必须限制分区，严禁全表扫描。
* **数据倾斜防御：** 大表 `GROUP BY` 或 `JOIN` 遇到大批量 `NULL` 值必须加随机数打散处理。
* **防御性编程：** 金额计算用 `COALESCE(amt, 0)` 兜底；除法运算的除数用 `NULLIF(divisor, 0)` 保护，避免报错阻断任务。
