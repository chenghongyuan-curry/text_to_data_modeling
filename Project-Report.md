# Text-To-Report：AI 驱动的自动化数仓开发引擎

## 1. 项目愿景：解放数据生产力

在数字化转型的浪潮中，数据已成为企业的核心资产。然而，传统的数据仓库开发模式却日益成为释放数据价值的瓶颈。**Text-To-Report** 项目应运而生，其核心愿景是：**将数据分析师和业务人员从繁琐性 ETL/SQL 编码中解放出来，让他们能够用自然语言与数据直接“对话”，从而实现从需求到报表的端到端自动化。**

我们旨在打造一个能理解业务、懂数据、会建模处 AI 数仓工程师，将过去数天甚至数周的开发周期，缩短到分钟级别。

## 2. 核心挑战：传统数仓开发的痛点

传统的数据开发流程通常涉及多个角色（业务分析师、数据产品经理、数仓工程师、ETL 工程师），链路长、沟通成本高、交付周期慢。

| **模型设计** | - **人工设计**: 架构师设计 ODS/DWD/DWS/ADS 模型。<br>- **血缘梳理**: 手动追溯数据来源。<br>- **文档编写**: 维护数据字典和模型文档。<br>- **时间成本**: 2-5 天 | - **AI 自动建模**: AI 基于元数据和需求，自动规划分层模型。<br>- **血缘自动发现**: 通过向量检索和图分析，秒级构建血缘。<br>- **元数据自同步**: 代码生成后，AI 自动回写元数据。<br>- **时间成本**: < 5 分钟 | 效率提升 **>98%** |

## 3. Text-To-Report 解决方案：智慧与架构的结晶

Text-To-Report 通过一套精密设计的自动化架构，实现了从业务语义到工业级 SQL 代码的跨层映射。

### 3.1 核心架构：需求驱动的“分层解构”流水线
项目采用了基于“执行计划”的模块化流水线 (`AutoDWPipeline`)。与传统 SQL 生成器不同，它不直接跳到最终代码，而是先生成“处理计划”，再驱动各层 Agent 协同工作：

1.  **感知与增强 (_step_knowledge_retrieval)**: 并非简单的关键词搜索，而是通过“语义扩展”将业务需求转化为技术标签，利用向量库检索并激活相关的元数据上下文。
2.  **解构与规划 (_step_analyze)**: AI 模拟首席架构师，将需求拆解为 `layer_plan`。系统会自动分析并准确识别表的生命周期状态（`new` 或 `existing`），并详细输出 DWD/DWS 等各层级内宽表的拼接与血缘组成逻辑。借此判定每一层是“复用现有模型”还是“增量补全字段”，形成一份指导后续代码生成的权威执行蓝图。
3.  **分层并发生成 (_step_generate_...)**: 根据规划，流水线依次触发 ODS 同步 Agent、DWD 建模 Agent 以及 ADS 服务 Agent。
4.  **闭环确认与纠偏 (Interactive Loop)**: 系统在每一层级都设置了“人工干预哨点”。用户可以针对 DWD 的关联逻辑或 ADS 的指标口径实时提出反馈，流水线会根据反馈自动执行局部重构。

### 3.2 知识库：多维语义锚点与“邻居发现”机制
数据仓库的灵魂在于元数据，Text-To-Report 的知识库（`KnowledgeManager`）不仅是存储，更是数据的“语义中枢”：

-  **多维向量化索引 (Multi-Dimensional Embedding)**: 
    - **技术语义**: 索引库名、表名、字段名。
    - **业务语义**: 索引字段级业务注释（Comment）及表级逻辑摘要（Logic Summary）。
    - **组织语义**: 索引业务组标识（Group），如“采购单元-po”。
-  **邻居发现 (Sibling Discovery)**: 这是一个创新的 RAG 增强机制。当 AI 检索到某张表时，系统会自动探测其所属的物理数据库或逻辑业务组，将该组下的所有相关表（如订单主表对应的支付、退款、物流邻居表）一并加载，彻底解决了传统 RAG 因语义稀释导致的“关联表丢失”难题。
-  **血缘自动补全**: 系统会解析元数据中的 `base_table`，自动为选中的 DWD 模型拉取其依赖的所有 ODS 源表，确保 AI 拥有“全景血缘视野”。

### 3.3 AI 引擎：注入数仓专家的“思维链”内核
AI 引擎 (`AutoDWEngine`) 的核心竞争力在于其内置了数仓领域的**逻辑强约束**，将专家的建模经验转化为算法的确定性：

1.  **动态路由 (Dynamic Routing)**: 引擎具备“关键词敏感度”。当识别到“明细/全量”时，自动强制表名使用 `dws_` 前缀；识别到“汇总/统计/TopN”时，切换至 `ads_`。这保证了数仓在初期就具备清晰的物理层级。
2.  **模型收敛规则 (Model Convergence)**: 这是解决“烟囱式开发”的关键。引擎强制执行“一业务组一模型”原则。如果多个源表同属一个 `Group`，AI 会被禁止生成多个 DWD 镜像，转而强制设计一个包含 `JOIN` 或 `UNION` 逻辑的收敛宽表模型。
3.  **分层强约束 (Layered Constraint)**: 在 SQL 生成阶段，引擎锁死了数据流向。ADS/DWS 的代码被禁止从 ODS 取数。如果所需字段不在 DWD 中，引擎会先触发 DWD 的结构升级，再由 ADS 引用更新后的 DWD 结构。这种“分层隔离”从根本上保证了数仓的可维护性。

## 4. 核心技术矩阵：Text-To-Report 的技术护城河

### 4.1 知识库与语义增强层 (Semantic Enhancement Layer)
- **知识向量化检索**: 不仅是搜关键词，而是将表/字段的 **业务注释、业务单元标识、物理血缘** 进行多维向量化，存储于 Milvus。
- **语义增强层 (Semantic Enhancement)**: 在检索前执行关键词联想，将业务词（如“订单头”）自动转译为技术关键词（header, po），解决语义断层。
- **RAG 模拟器 (Sibling Discovery)**: 模拟资深 DBA 思路，当命中一张 ODS 表时，自动“感知”并拉取同一业务单元下的邻居表，确保主从表不失联。

### 4.2 动态血缘与元数据管理 (Dynamic Lineage)
- **动态血缘**: 基于元数据动态推导支撑新指标所需的依赖路径。系统反向解析 SQL，实时更新 JSON 元数据名。
- **元数据即管理**: 每一行生成的代码都会被反向解析，自动更新 JSON 元数据，实现“元数据即代码”。

### 4.3 AI 引擎与思维链 (Chain of Thought, CoT)
- **思维链输出**: 系统在分析需求时，必须遵循：`判定路由 -> 锁定业务单元 -> 执行收敛 -> 强分层校验` 的思维链条。
- **标准化命名路由**: 在分析规划阶段严格注入全层级（ODS/DWD/DIM/DWS/ADS/TMP）的命名铁律（如附带全增量标识 `df/di`），确保表名自带业务元属性。
- **SQL 模式库 (Pattern Library)**: 内置 `AliOrcSerDe`, `DistributedMergeTree` 等生产级 SQL 模板，确保生成的代码具有“大厂味道”。

### 4.4 质量守卫：基于样本库的“好/坏味道”约束
为了确保 AI 生成的代码不仅“能跑”，而且“优雅”，系统引入了基于 Few-shot Prompting 的双向样本库：

*   **“好味道”库 (Standard Pattern Library - 优秀规范与范式)**: 
    引擎内置了大量高标准的 SQL 与 DDL 样例作为 AI 的学习蓝本。
    - **结构化设计**: 强制使用 **CTE (Common Table Expressions)** 代替多层嵌套子查询，极大增强逻辑的可读性与维护性。
    - **工业级标准**: 针对 ClickHouse 生产环境，固化了分布式表与本地表的 DDL 模板，包含 TTL 与 Index 最佳实践。
    - **逻辑严密与防御性编程**: 包含显式的类型转换、统一的枚举转码规范，强制规定金额等指标使用 `COALESCE` 兜底、除法语局使用 `NULLIF` 防护报错。在大表关联时融合了随机数打散等数据倾斜防御的工业级手段。
    - **强审计与数据规范**: 引擎强制统管建表及 DML 语句中的分区行为（如 `_pt` 默认赋值 `toDate(now())`），彻底杜绝 DDL 与解析脚本之间的分区割裂；强制模型表预埋并维护 `create_time` 和 `modify_time` 审计字段；同时严格约束各层级表和字段的注释覆盖率须达到 100%。
*   **“坏味道”库 (Anti-Pattern Guardrail - 错误反例规避)**: 
    通过负向约束（Negative Constraints）明确禁止以下技术债模式，一旦发现立即触发自愈流程：
    - **架构越权**: 绝对禁止 `ADS/DWS -> ODS` 的跨层调用样例，杜绝“层级穿透”现象。
    - **性能隐患**: 禁止使用 `SELECT *`，禁止在 DDL 中遗漏分区键，禁止不带分区过滤的全表扫描。
    - **逻辑漏洞**: **强制拦截笛卡尔积 (Cartesian Product)**。系统通过语法解析器深度检查 `JOIN` 子句，严禁出现无关联条件的全量笛卡尔积查询，从源头杜绝线上“慢查询炸弹”。
    - **语义污染**: 禁止在 SQL 的任何环节出现中文命名或硬编码业务参数，确保系统纯洁性。

## 5. 核心应用场景
### 5.1 场景一：新增报表
- **用户需求**: “我需要一张采购订单明细报表，要包含订单头和订单行的所有字段，并且状态需要转成中文。” 
- **Text-To-Report 执行流程**: 
  1.  **AI 分析**: 识别出关键词“明细报表”，判定目标层为 `DWS`。识别出实体“采购订单头”、“采购订单行”。
  2.  **RAG 检索**:
     - 分别检索“订单头”和“订单行”，同时命中 `ods_sy_order_db_db_order` 和 `ods_sy_order_db_db_order_item`。
     - 发现这两个表同属 `po` 业务组。
  3.  **AI 建模**:
     - **DWD 层**: 判定需要将这两个 ODS 表合并，规划出**一个新**的 DWD 模型 `dwd_po_order_df`。
     - **DWS 层**: 规划出最终的报表 `dws_po_order_detail_df`，其数据源指向 `dwd_po_order_df`。
  4.  **代码生成**:
     - 自动生成 `dwd_po_order_df.sql` (包含 `JOIN` 和 `CASE WHEN` 状态转码)。
     - 自动生成 `dws_po_order_detail_df.sql` (从 DWD 选出所有字段)。
     - 自动生成对应的 ClickHouse DDL 文件。
  5.  **元数据回写**: 将新生成的 DWD 和 DWS 表的结构、血缘、逻辑反向解析并存入 `metadata/` 目录，为下一次开发做准备。

### 5.2 场景二：老报表增加字段
- **用户需求**: “在采购订单明细报表里，增加一个‘创建人姓名’字段。”
- **Text-To-Report 执行流程**: 
    1.  **AI 分析**: 识别出目标是修改现有报表 `dws_po_order_detail_df`。
    2.  **RAG 检索**:
        - 检索“创建人姓名”，未命中该字段。
        - 加载 `dws_po_order_detail_df` 及其依赖 `dwd_po_order_df` 的元数据。
    3.  **AI 建模 (字段溯源)**:
        - **DWS 层**: 发现 `dws_po_order_detail_df` 缺少“创建人姓名”。
        - **DWD 层**: 追溯到 `dwd_po_order_df`，发现它也缺少该字段。
        - **ODS 层**: 发现 `ods_sy_order_db_db_order` 中有 `creator`，但是为Int类型说明是id，匹配关联关系,发现 `ods_platform_user_user` 中有 `name`。
    4.  **AI 规划**:
        - 判定需要**更新** `dwd_po_order_df`，增加与 `ods_platform_user_user` 的关联逻辑。
        - 判定需要**更新** `dws_po_order_detail_df`，从更新后的 DWD 中 `SELECT` 出新字段。
    5.  **代码生成**: 生成更新后的 `dwd_...sql` 和 `dws_...sql`。
    6.  **更新元数据**: 将更新后的 DWD 和 DWS 表的结构、血缘、逻辑反向解析并存入 `metadata/` 目录，为下一次开发做准备。
